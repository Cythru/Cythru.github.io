APEX Linux Edition: Building a Fully Modular Linux OS in Rust, C++, and Zig

How we refactored a bare-metal kernel into a modular Linux-based OS — and why the architecture matters.

Introduction

APEX Linux Edition v0.2.0 “Forge” is out. It represents a complete architectural pivot — from a custom bare-metal kernel (v0.1.0 “Genesis”) to a fully modular Linux-based operating system built with Rust, C++, and Zig. This is not a cosmetic change. Every subsystem was redesigned, every interface was rethought, and the result is something fundamentally different from what we started with.

This post covers why we made the switch, how the module system works at a technical level, what each of the 19 modules does, how the gaming layer integrates Steam and Proton without friction, and how a single Zig build command orchestrates Rust, C++, and Zig code into a bootable ISO. If you care about systems architecture, this is the deep dive.

Why move off bare-metal?

The bare-metal kernel was a great learning exercise. We wrote our own bootloader, our own memory manager, our own interrupt dispatcher. It worked. But there were real costs that became harder to justify as the project matured.

We were reimplementing things that Linux already does correctly — process scheduling, virtual memory management, driver ecosystems, hardware compatibility tables. Every week we spent writing another PCI enumeration routine or USB host controller driver was a week we were not spending on the things that actually differentiate APEX: the module system, the gaming layer, the snapshot engine, the package manager.

The insight was simple: the value is not in owning every layer. It is in owning the right layers. Linux gives us a battle-tested kernel with decades of driver support, a stable syscall ABI, and a security model that has been audited by thousands of engineers. We take that foundation and build everything above it — PID 1, the display server, the gaming stack, the network manager, the package system, the update mechanism. Every layer the user interacts with is ours.

The bare-metal kernel still exists in the repository history. It was an important phase. But APEX is no longer a kernel project — it is an operating system project, and that distinction matters.

The module system

Every component in APEX is a self-contained module. There are no monolithic daemons, no tightly coupled subsystems, no hidden dependencies between components. Each module is a standalone binary with a well-defined interface, declared through a manifest file called apex_module.toml.

The manifest declares everything the module system needs to know: what the module provides, what it requires, which other modules it depends on, what Linux capabilities it needs, and how to build it. Here is a real example from the gaming module: 

[module]
name = "gaming"
version = "0.2.0"
description = "Steam/Proton/Gamescope integration"
language = "rust"
category = "desktop"

[capabilities]
provides = ["steam-integration", "proton-management",
            "gamescope", "performance-tuning", "overlay"]
requires = ["linux-kernel >= 6.0"]

[dependencies]
modules = ["init", "display"]

[build]
type = "rust-bin"
entry = "src/main.rs"

[runtime]
user = "root"
capabilities = ["CAP_SYS_NICE", "CAP_SYS_ADMIN",
                "CAP_NET_ADMIN"]

The module system itself (the module crate) is responsible for loading, registering, and managing all other modules. It implements an ELF64 module loader that handles relocations (R_X86_64_64, R_X86_64_PC32, R_X86_64_PLT32, R_X86_64_GOTPCREL, R_X86_64_32S), maintains a symbol table of up to 16,384 entries, and supports up to 256 simultaneously loaded modules.

Crucially, the module ABI is stable and versioned. The ApexModuleDescriptor struct is defined in a C header (include/apex/module.h), which means modules can be written in any language that can produce ELF shared objects with C-compatible symbols — Rust, C++, Zig, plain C, anything. This is by design. We do not want the module system to be language-locked. Today, all 19 modules happen to be written in Rust, but the C++ drivers and Zig utilities link through the same ABI without special handling.

Hot-reload is supported at the module level. You can unload and reload a module at runtime without restarting the system. The module system tracks dependency graphs and prevents you from unloading a module that other running modules depend on. This is not theoretical — it is how we do development. Change a module, rebuild it, hot-reload it, test it, all without rebooting.

The 19 modules

APEX ships with 19 modules organized into three tiers: Core, System, and Desktop. The tiers reflect boot order and dependency direction — Core modules start first and have no dependencies on higher tiers. System modules depend on Core. Desktop modules depend on both.

DESKTOP [ user-facing services ] display gaming audio network compat ai www re_suite SYSTEM [ OS plumbing ] init session snapshot security packages update hardware CORE [ kernel-adjacent primitives ] vfs ipc exec net_stack module ──────────────────────────────────────────────────── LINUX KERNEL 6.12 [ scheduler, mm, drivers, fs ]

Core tier — These are the foundational primitives. vfs provides the virtual filesystem layer. ipc handles inter-process communication between modules. exec manages process execution and lifecycle. net_stack provides the network stack abstraction. module is the module loader itself — the only module that bootstraps without going through the module system.

System tier — These are the operational services. init is PID 1 — it brings up the system, manages service dependencies, and handles shutdown. session manages user sessions and seat tracking. snapshot provides copy-on-write system snapshots for instant backup and restore. security handles MAC policies and sandboxing. packages is the apx package manager with full dependency resolution. update handles OTA updates with delta patches and automatic rollback. hardware manages device detection, driver binding, and hardware abstraction.

Desktop tier — These are the user-facing services. display is the Wayland compositor. gaming handles Steam, Proton, and Gamescope. audio manages PipeWire audio routing. network provides NetworkManager-level network configuration. compat handles compatibility layers for non-native applications. ai provides on-device inference capabilities. www implements the APEX-WWW distributed protocol. re_suite is the built-in reverse engineering toolkit with disassembler, kprobes, and execution tracer.

Gaming-first design

APEX is, unambiguously, a gaming-first OS. The gaming module is not a bolt-on afterthought — it is a first-class citizen with deep integration into the display, hardware, and performance subsystems.

The Steam integration works by parsing Steam’s VDF (Valve Data Format) library files directly. APEX reads libraryfolders.vdf to discover game install locations and appmanifest_*.acf files to enumerate installed titles. There is no dependency on the Steam client being running for game discovery — APEX can inventory your library from cold storage.

Proton management goes beyond just launching games through Wine. The gaming module configures the full Proton environment per-game: STEAM_COMPAT_DATA_PATH, STEAM_COMPAT_CLIENT_INSTALL_PATH, WINEPREFIX, WINEESYNC, WINEFSYNC, DXVK HUD toggles, VKD3D configuration, and per-title environment overrides. It manages Proton version selection, prefix initialization, and DXVK cache directories. When you launch a game, the gaming module builds the exact environment that game needs and hands it off to Proton with zero user configuration required.

Gamescope integration provides per-game display isolation. Each game runs inside its own nested Wayland compositor with configurable resolution scaling, refresh rate targeting, and AMD FSR (FidelityFX Super Resolution) profiles. The gaming module generates Gamescope launch parameters based on the game’s profile — internal resolution, output resolution, FSR sharpness, frame limiter settings, and HDR flags. This means you can run a game at 720p internal with FSR upscaling to 1080p at a locked 60fps, all managed automatically.

Performance tuning runs alongside the gaming stack. The gaming module manages CPU governor switching — flipping to performance when a game launches and back to powersave when it exits. It handles GPU clock management on AMD hardware through sysfs interfaces, configures TDP limits on supported hardware, and generates MangoHud overlay configurations for real-time performance monitoring. The overlay config includes FPS, frametime graphs, CPU and GPU utilization, VRAM usage, and temperature readouts — all generated from a declarative profile rather than requiring users to hand-write MangoHud config files.

Build system: Zig orchestrating everything

The APEX build system is one of the more unusual aspects of the project. The top-level build.zig file is the single entry point for the entire OS build. It does not just build Zig code — it orchestrates everything.

Here is what happens when you run zig build iso:

  • Zig compiles its own utilities (utils.zig and any Zig-native tools)
  • Zig invokes Cargo to build all 19 Rust modules — each module is a separate crate in modules/, and Cargo resolves their internal dependency graph through a workspace Cargo.toml
  • Zig invokes CMake to build the C++ drivers in drivers/ — hardware abstraction code, GPU kernel module shims, and low-level device interfaces
  • All build artifacts are collected and linked according to the module manifests
  • The rootfs is assembled: directory structure, module binaries, configuration files, default package database, and boot configuration
  • A bootable ISO image is produced with the Linux kernel, initramfs, and the complete APEX userspace

One command. No intermediate steps. No shell script orchestration. Zig’s build system is powerful enough to invoke external build tools as build steps, track their inputs and outputs for incremental compilation, and parallelise independent build targets across all available cores. The result is a build system that is both reproducible and fast — a clean build from source to bootable ISO takes minutes, not hours.

The choice of Zig as the build orchestrator was deliberate. We considered Make, CMake, Meson, and Bazel. Make and CMake lack the expressiveness to describe this kind of polyglot build cleanly. Meson comes close but does not handle Rust workspaces well. Bazel could do it but brings enormous complexity. Zig’s build system is a real programming language (Zig itself) with first-class support for invoking system commands, declaring dependencies between steps, and producing artifacts — and it comes free with the Zig compiler we were already using for the utility code.

Snapshot and rollback

The snapshot module implements copy-on-write system snapshots. Before any system update, package installation, or configuration change, APEX automatically creates a snapshot of the current system state. If something goes wrong, you can roll back to any previous snapshot instantly.

This is not filesystem-level snapshotting like Btrfs subvolumes (though APEX can use Btrfs). The snapshot module operates at the APEX layer — it captures the state of installed packages, module configurations, system settings, and boot configuration. A rollback restores all of these atomically. The update module integrates directly with snapshots: every OTA update creates a pre-update snapshot, applies delta patches, verifies the result, and only commits the update if verification passes. If verification fails, the pre-update snapshot is restored automatically. Zero manual intervention.

The reverse engineering toolkit

The re_suite module is unusual for an operating system — it ships a built-in reverse engineering toolkit. This includes a disassembler capable of decoding x86-64 instruction streams, a kprobes interface for dynamic kernel tracing, an execution tracer that records syscall sequences and memory access patterns, and a binary analysis framework for inspecting ELF binaries.

Why ship this in an OS? Because we believe that users should be able to understand what their system is doing at every level. The RE toolkit is not a security afterthought — it is a first-class diagnostic tool. If a game is crashing through Proton, you can trace the exact syscall sequence that diverges from expected behavior. If a module is misbehaving, you can attach kprobes to its hot paths without recompiling anything. Transparency is not optional.

The APEX-WWW distributed protocol

The www module implements APEX-WWW, a distributed protocol designed for peer-to-peer content distribution and decentralized service discovery. While still in early development, the protocol is already used internally for distributing package metadata across APEX instances. The long-term vision is a fully decentralized alternative to traditional client-server web architecture — content-addressed, cryptographically verified, and resistant to single points of failure.

This is experimental. We are not claiming it replaces HTTP tomorrow. But the infrastructure is there, it works, and it ships as a module you can enable or disable without affecting anything else.

What’s next

The v0.2.0 “Forge” release establishes the modular foundation. The roadmap for the next series of releases focuses on four areas:

  • SMP multicore scheduler integration — Leveraging Linux’s multicore scheduling with APEX-specific task priority hints from the gaming and performance modules
  • ACPI power management module — Fine-grained power state control, suspend/resume hooks for all modules, and battery-aware performance profiles
  • ApexFS custom filesystem — A purpose-built filesystem optimized for gaming workloads: fast random read, aggressive prefetch, and transparent compression for game assets
  • Public package registry — An official APEX package registry with signed packages, automated builds, and community contributions

The source is public. The development is in the open. If you want to follow along, contribute, or just read the code, the repository is at github.com/Cythru/apex-kernel.
